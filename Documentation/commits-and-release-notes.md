# Commits, Changelogs, Release Notes

## 1. The "release notification system" in one paragraph

Commits are written for developers and follow strict Conventional Commits. A per-component changelog is auto-generated from commits at every release. Developers, integrators, and users tracking specific tickets read it. A blog post is curated at release moments worth announcing (mender.io). Customers and prospects read it. Feature documentation pages in the regular docs (docs.mender.io) carry status badges (Preview / Beta / GA (default) / Deprecated) and version requirements, edited as part of normal documentation work.

## 2. Three artifacts, three audiences

| Artifact | URL pattern | Audience | Update model |
|---|---|---|---|
| **Per-component changelog** | `docs.mender.io/release-information/<component>/<version>` | Developers, integrators, users tracking a ticket | Auto-generated by git-cliff at release and curated at release prep; subsequent edits limited to late-disclosure security entries |
| **Blog announcement** | `mender.io/blog/<slug>` (tag: `product-news`) | Customers, prospects, marketing reach | Curated at release moments worth announcing |
| **Feature documentation** | `docs.mender.io/<feature-area>/...` (existing docs) | Users learning how to use a feature | Edited as part of normal documentation work; status badge + requirements section |

## 3. Commit rules

### 3.1 Format

Strict [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/), enforced in CI:

```
type(scope): subject

body (optional, can be multiple paragraphs)

Trailer: value
Trailer: value
```

**Recommended:**
- Lowercase type from the allowed set
- `(scope)` optional but encouraged
- Subject with no trailing period
- Imperative mood ("add", "fix", "remove"; not "added", "fixes", "removed")

- We are **not** blocking CI on behavioral framing, only on format. Framing is a code-review norm with reference examples.

**Scope policy:** Scope identifies the affected module, component, microservice, or functionality area, e.g., `(server)`, `(client)`, `(gui)`, `(deployments)`, `(inventory)`, `(iot-manager)`. Conventions:

- Free-form by default (no global allowlist), but **individual repos may enforce their own rules** - e.g. `mender-server` defines an allowed-scope list and requires a non-empty scope on `feat`/`fix` via its `commitlint.config.js`. Follow the local repo's linter.
- Lowercase, single token; hyphenated if needed (`(iot-manager)`).
- Match the directory, service name, or feature area where possible. Prefer the specific service/component (`(deployments)`) over a broad repo-level scope (`(server)`) where the repo distinguishes them.

**Allowed types:** `feat`, `fix`, `perf`, `refactor`, `chore`, `ci`, `build`, `docs`, `test`, `style`, `revert`.

**Examples — format:**

| | Subject |
|---|---|
| ❌ | `MEN-9415: Trim large deployment logs` |
| ✓ | `feat(deployments): trim deployment logs that exceed the size limit` *(ticket goes in `Ticket:` trailer)* |
| ❌ | `FIX: Prevent segfault by correctly handling dbus restarts` |
| ✓ | `fix(client): prevent crash when dbus restarts unexpectedly` |
| ❌ | `Fixes nil pointer dereference` *(no type)* |
| ✓ | `fix(server): prevent crash when device authenticates with expired token` |

### 3.2 Behavioral framing for `feat` and `fix`

**Subjects for `feat` and `fix` describe the behavior change, not the code change.** This is for developer benefit primarily. Behavioral framing makes `git blame` and bisect more useful, and ages better than implementation-speak. It also happens to make the auto-generated changelog more readable for users, which is a bonus.

If a commit has no externally observable behavior change, the type is wrong. In this case use `refactor:`, `perf:`, or `chore:`.

**Observable doesn't mean visible to all users.** A bug that silently corrupted data is observable when fixed, even if no UI changes. A subtle race condition is observable when its symptom (wrong status, lost message, double-execution) stops happening. The test is whether *anything externally noticeable* changes, not whether a user staring at the UI would catch it. If literally no externally observable behavior changes, the type isn't `fix:` or `feat:`.

**Examples — feat:**

| | Subject |
|---|---|
| ❌ | `feat(client): add retry logic to inventory module` |
| ✓ | `feat(client): automatically retry failed inventory reports` |
| ❌ | `feat(server): implement endpoint for software listing` |
| ✓ | `feat(server): expose installed software per device via API` |

**Examples — fix:**

| | Subject |
|---|---|
| ❌ | `fix(server): fix nil pointer in device auth handler` |
| ✓ | `fix(server): prevent crash when device authenticates with expired token` |
| ❌ | `fix(gui): fixed an issue that prevented requiring confirmation on deployment creation` |
| ✓ | `fix(gui): restore confirmation prompt before creating deployments` |
| ❌ | `fix(gateway): update gRPC error mapping` |
| ✓ | `fix(gateway): return correct error codes for rejected deployments` |
| ❌ | `fix(server): fix race in deployment status polling` *(implementation-speak: "race" and "polling" are engineering terms; a user reads "race" and doesn't know what they experience)* |
| ✓ | `fix(server): show consistent deployment status during concurrent updates` |

**Examples — `refactor`/`chore` are fine to be implementation-focused:**

| | Subject | Why |
|---|---|---|
| ✓ | `refactor(gui): migrate deployment creation dialog to RHF` | Refactor - implementation detail is appropriate |
| ✓ | `chore(gui): update test snapshots` | Chore - no user impact, no need to dress it up |
| ✓ | `perf(server): cache auth tokens to reduce DB load on hot path` | Perf - implementation framing OK; behavioral framing better when measurable |

### 3.3 Type selection

`chore:` is the most-abused type. Narrow it:

- Touches product code → not `chore`. Use `refactor:`, `perf:`, `fix:`, or `feat:`.
- Dependency bumps → `chore: bump` or `chore(deps):` (cliff.toml reclassifies these as Security).
- Repo housekeeping, release commits → `chore:` is appropriate.
- Internal-only changes that shouldn't appear in any changelog → use the `(internal)` scope (already filtered).

**Examples:**

| | Subject |
|---|---|
| ❌ | `chore: change auth caching strategy` *(touches product behavior)* |
| ✓ | `perf(server): cache auth tokens to reduce DB load on hot path` |
| ❌ | `chore: rename ReleaseOrImageFilter to ReleaseFilter` *(touches product code)* |
| ✓ | `refactor(server): rename ReleaseOrImageFilter to ReleaseFilter` |

### 3.4 Breaking changes

Per the [Conventional Commits spec](https://www.conventionalcommits.org/en/v1.0.0/#specification), breaking changes can be indicated by **either**:

- A `!` marker after the type/scope: `feat(api)!:`, `fix(client)!:`, `refactor(server)!:`
- A `BREAKING CHANGE:` footer with the description

Both forms are accepted. The `!` marker is more visible in `git log --oneline`; the footer carries more migration detail. Use them together when the change warrants it.

**Rendering:** cliff.toml emits `**BREAKING**: <description>` inline within the changelog entry for any commit carrying either the `!` marker or a `BREAKING CHANGE:` footer.

**Example:**

```
feat(api)!: remove legacy /devices/list endpoint

Removed in favor of /devices/v2/list which supports pagination
and filtering. The legacy endpoint had been deprecated since 4.1.

BREAKING CHANGE: Clients calling /devices/list must migrate to
/devices/v2/list. The legacy endpoint now returns 404. See the
migration guide at https://docs.mender.io/migration/devices-v2.

Ticket: MEN-9420
Signed-off-by: Jane Developer <jane.developer@northern.tech>
```

### 3.5 Body usage

Body is for engineering detail - root cause, migration notes, design rationale, references. **By default, the body is NOT rendered in the changelog** - only the subject line is. The body lives in `git log` for developers; the changelog stays concise.

To opt the body into the changelog, add a `Changelog: Commit` trailer. To replace the rendered line with a different user-facing sentence, use `Changelog: <sentence>`. To skip the commit from the changelog entirely, use `Changelog: None`.

**Use the body when:**
- The subject doesn't explain why the change was needed
- A reviewer would benefit from knowing the root cause or alternatives considered
- There are non-obvious implications for downstream code
- Migration steps are needed for breaking changes

**Skip the body when:**
- The subject is fully self-explanatory
- The commit is mechanical (dependency bump, generated code update)

The rationale for hiding bodies by default: well-written subjects already carry the customer-relevant information; bodies tend to add engineering noise to the changelog. When a body genuinely belongs in the customer-facing changelog, opt in explicitly with `Changelog: Commit`.

### 3.6 Trailer reference card

All trailers are optional except where noted.

| Trailer | Purpose | Example |
|---|---|---|
| `Ticket:` | Link to Jira (auto-linked) | `Ticket: MEN-9420` |
| `Changelog: None` | Skip this commit from the changelog entirely | `Changelog: None` |
| `Changelog: Commit` | Include the commit body in the changelog (default is subject only) | `Changelog: Commit` |
| `Changelog: <sentence>` | Override the rendered line with a user-facing sentence | `Changelog: Devices now retry failed inventory reports automatically` |
| `Deprecation: <sentence>` | Populate the Deprecations section | `Deprecation: The --legacy-auth flag is deprecated; use --auth-mode=token.` |
| `BREAKING CHANGE: <text>` | Migration detail for breaking commits | See section 3.4 |
| `Signed-off-by:` | Managed by tooling, required on every commit | `Signed-off-by: Jane Developer <jane@northern.tech>` |

**Cherry-pick annotation:** `git cherry-pick -x` appends `(cherry picked from commit <hash>)` to the body for backport traceability. This is a git-inserted parenthetical, not a trailer in the `Token: value` sense.

**No `Changelog:` trailer = default.** Absent any `Changelog:` trailer, the commit is rendered with subject only. Add `Changelog: Commit` to include the body, `Changelog: <sentence>` to override the line, or `Changelog: None` to skip the commit.

**Deprecation vs. Breaking Change.** Deprecation announces upcoming removal; the deprecated thing still works today. Use `Deprecation:` to surface the warning. Use `!` / `BREAKING CHANGE:` only when behavior actually changes today (e.g., an endpoint that used to work now returns 404, a flag that used to be accepted now errors).

**IMPORTANT: trailer values are case-sensitive.** Normalize on **uppercase initial** for special values: `Changelog: None`, `Changelog: Commit`, `Ticket: None`. Lowercase forms (`changelog: commit`, `Changelog: none`) are silently ignored by [cliff.toml](https://github.com/mendersoftware/mendertesting/blob/master/utils/cliff.toml) because the template compares against the exact uppercase strings. Free-text values (`Changelog: <sentence>`) are not case-sensitive.

**Trailer examples:**

| | Trailer |
|---|---|
| ❌ | `Changelog: None` on a `feat:` *(if no user impact, the type is wrong — should be `refactor:` or `chore:`)* |
| ❌ | `Changelog: Commit` on a `feat:` where the body is implementation-speak *(fix the body first, or skip the trailer)* |
| ✓ | `Changelog: Commit` on a `fix:` where the body explains customer-visible impact worth surfacing |
| ✓ | `Changelog: None` on a `chore:` that touches a generated file no customer cares about |

### 3.7 When the type turns out to be wrong

Git is a blockchain; original commits don't move. But sometimes a commit's type looks correct at write time and wrong in retrospect. The most common cases:

- A change written as `feat:` is later backported, suggesting it should have been (or now functions as) a `fix:`. Real example: [mendersoftware/mender@eb1a780d](https://github.com/mendersoftware/mender/commit/eb1a780d) — committed as `feat:`, later considered as backport material.
- A change written as `fix:` turns out to have user-visible impact customers will want to know about (more `feat`-like).
- A `chore:` later proves to have customer-visible consequences someone wants surfaced.

**The original commit stays.** Don't rewrite history. Don't amend types after the fact.

**Recovery paths:**

1. **For backports:** the cherry-pick on a release branch is a new commit and gets its own type that reflects what it does *on that branch*. If the original was `feat:` on master and you cherry-pick to a patch release, the backport is `fix:`. Both commits are honest about their context.
2. **For the published changelog:** the release-prep curation step (§4.2 / §4.1) can move an entry to a different section, rewrite the rendered line, or promote a misclassified change into Highlights. The git log keeps the original; the customer-facing changelog reflects the right intent.

**When in doubt at write time:** prefer `fix:` over `feat:` for borderline changes that read more like restoring/improving existing behavior than adding new capability. Backports stay cleaner and customer messaging can always promote a `fix:` to highlight-worthy if it turns out to be significant.

### 3.8 When a `Ticket:` is expected

The `Ticket:` trailer is never enforced by the linter. It's a code-review norm, and the expectation depends on the change and the repository:

| Commit kind | Ticket |
|---|---|
| Breaking change (`!` or `BREAKING CHANGE:`) | Expected and reviewers should ask for one |
| `feat:` / `fix:` | Expected where the repo tracks work in Jira; not expected in repos whose changes are routinely ticketless (e.g. shared configs, tooling) |
| Everything else | Optional |

Use common sense: add a `Ticket:` when the change has a related ticket or warrants one. A repository's own history defines the local norm; reviewers hold the line.

### 3.9 External (and ticketless) contributions

The only hard, CI-enforced requirement is the **Conventional Commits format** (§3.1). Everything
beyond that is not a gate: the `Ticket:` trailer is never enforced anywhere (§3.8), and
scope is only checked where a repo's own linter checks it. None of it blocks a
contribution - internal or external.

We don't expect external contributors to learn our trailer conventions. Maintainers may amend or fix up commit messages in the PR before merging (we already touch them to add `Signed-off-by:`). Anything we genuinely require of contributors belongs in the repo's contribution guide / PR checklist, not buried in this policy.

## 4. Per-component changelog structure

The changelog is generated from commits by git-cliff (using [our cliff.toml](https://github.com/mendersoftware/mendertesting/blob/master/utils/cliff.toml) centrally maintained in `mendertesting`), triggered at release time by [release-please](https://github.com/googleapis/release-please) which also drives version bumps and opens release PRs. The output then goes through a short curation pass at release prep before publishing. Curation covers Highlights (hand-written), Security (CVE refs added), and Improvements (refactor promotion). Other sections are auto-generated.

Lives at `docs.mender.io/release-information/<component>/<version>`.

### How `Changelog:` trailers render

For a commit like:

```
feat(deployments): trim deployment logs that exceed the size limit

Logs over 1 MiB are truncated to the last 64 KiB before sending
so the deployment server isn't asked to swallow arbitrarily large
payloads from misbehaving devices.

Ticket: MEN-9415
```

The rendered changelog entry depends on the `Changelog:` trailer:

| Trailer | Rendered line |
|---|---|
| *(no `Changelog:` trailer — the default)* | *(deployments)* trim deployment logs that exceed the size limit (MEN-9415) |
| `Changelog: Commit` | *(deployments)* trim deployment logs that exceed the size limit (MEN-9415)<br><br>&nbsp;&nbsp;Logs over 1 MiB are truncated to the last 64 KiB before sending so the deployment server isn't asked to swallow arbitrarily large payloads from misbehaving devices. |
| `Changelog: Devices report log tails when uploads would exceed the size limit.` | Devices report log tails when uploads would exceed the size limit. (MEN-9415) |
| `Changelog: None` | *(commit omitted from the changelog entirely)* |

### 4.1 Section order

1. **Highlights** *(manually curated - section 4.2)*
2. **Breaking changes** *(omitted if none)*
3. **Deprecations** *(omitted if none)*
4. **Security** *(omitted if none; curated; auto-includes CVE-flagged dependency bumps; manual entries added at release prep for coordinated disclosures or CVEs assigned post-commit)*
5. **New features** *(from `feat:`)*
6. **Improvements** *(from `perf:`; behavior-affecting `refactor:` can be promoted at release prep)*
7. **Bug fixes** *(from `fix:`)*
8. **Dependency updates** *(from `chore: bump`, `chore(deps):`; classified as Security)*
9. **All tickets resolved in this release** *(appendix - flat table)*

Most important / risky / actionable at the top. Empty sections are omitted.

### 4.2 Highlights section (curated)

Solves the readability problem of "50 consecutive GUI fixes" by collapsing related work under one feature header with a lead paragraph and ticket list.

**Owner:** release engineer + PO, written at release prep.
**Budget:** 15-30 minutes per release.

**Format:**

```markdown
## Highlights

### Phased deployments
Roll out updates to your fleet in controlled phases with percentage-based
device selection and configurable wait times between phases. Pause and
resume rollouts from the deployment dashboard.

[Documentation →](https://docs.mender.io/...)

### Server-side delta generation
Generate delta artifacts server-side, removing the need to run
mender-artifact locally and reducing storage requirements for fleets
with many devices.

[Documentation →](https://docs.mender.io/...)
```

The Highlights content is the source material for the blog post.

## 5. Blog and release announcements

### 5.1 When to publish

A blog post is published when a release moment is worth announcing:

- Major / minor releases of customer-visible components
- Phase changes for tracked features (Preview, Beta, GA)
- Bundled multi-component features going live

**Not** for patch releases unless something exceptional shipped. **Not** for component releases that ship only internal improvements.

**Decision owner:** the **Product Owner** decides whether a release moment warrants a blog post, in consultation with engineering and marketing.

### 5.2 Workflow

1. Pull the Highlights section(s) from the relevant per-component changelog(s).
2. Human (PO, marketing) edits the draft: tightens voice, adds screenshots if UI, ensures factual accuracy.
3. Publish at `mender.io/blog/<slug>` with tag `product-news`.
4. Update relevant feature docs (status badges, version requirements).

## 6. Feature documentation conventions

For features that span phases (Preview → Beta → GA → Deprecated) or matter to customers asking "do you support X?", the existing feature docs page gets two small additions:

**Status indicator** at the top of the page:

> **Status:** Preview *(or Beta / GA / Deprecated)*

Updated/removed as the feature progresses. Edited as part of normal doc work.

**Requirements section** stating minimum versions:

> **Requires:** Mender Server 4.4 or later, Mender Client 5.1 or later

Updated only when the floor genuinely moves (security baseline, EOL of a supported version); not on every release.
